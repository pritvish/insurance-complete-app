System Architecture

The platform uses an event-driven microservices architecture. Services communicate asynchronously via Kafka. The Workflow Orchestrator implements the Saga pattern to coordinate multi-step business processes spanning multiple services.

Transactional Outbox Pattern

📤
Services: customer, policy, payment, billing, claims
Solves the dual-write problem: DB write + Kafka publish atomically
Problem: If you write to DB and then publish to Kafka, the service can crash between the two operations. The event is lost. This is the dual-write problem.
Solution: Write to both the business table AND the outbox table in the same transaction. A background poller reads PENDING outbox events and publishes them to Kafka.
sequenceDiagram participant SVC as Service Layer participant DB as PostgreSQL participant PUB as OutboxPublisher\n(@Scheduled 100ms) participant KAFKA as Kafka SVC->>DB: BEGIN TRANSACTION SVC->>DB: INSERT INTO customers (...) SVC->>DB: INSERT INTO outbox_events (status=PENDING, payload=...) SVC->>DB: COMMIT Note over DB: Both writes atomic — no dual-write risk loop Every 100ms PUB->>DB: SELECT * FROM outbox_events WHERE status='PENDING' DB-->>PUB: [pending events] PUB->>KAFKA: publish(topic, payload) PUB->>DB: UPDATE outbox_events SET status='PROCESSED' end

OutboxEvent Entity

@Entity
@Table(name = "outbox_events")
public class OutboxEvent {
  @Id UUID id;
  String aggregateType;
  String aggregateId;
  String eventType;
  String payload;           // JSON
  OutboxStatus status;     // PENDING/PROCESSED/FAILED
  Instant createdAt;
  Instant processedAt;
}

OutboxPublisher

@Component
public class OutboxPublisher {

  @Scheduled(fixedDelay = 100) // 100ms poll
  public void publishPendingEvents() {
    outboxRepo.findByStatus(PENDING)
      .forEach(event -> {
        try {
          kafkaTemplate.send(event.getEventType(),
                             event.getPayload());
          event.setStatus(PROCESSED);
          event.setProcessedAt(Instant.now());
        } catch (Exception e) {
          event.setStatus(FAILED);
        }
        outboxRepo.save(event);
      });
  }
}

PolicyEventStore.appendEvent()

// Must be called within an existing transaction
@Transactional(propagation = MANDATORY)
public void appendEvent(String policyId,
    String eventType, Object payload, String correlationId) {

  long nextVersion = policyEventRepo
    .findMaxVersionByPolicyId(policyId)
    .orElse(-1L) + 1;

  PolicyEvent event = new PolicyEvent();
  event.setPolicyId(policyId);
  event.setEventType(eventType);
  event.setEventVersion(nextVersion);
  event.setPayload(toJson(payload));
  policyEventRepo.save(event);

  // Also save to outbox atomically
  outboxRepo.save(toOutboxEvent(event));
}

SagaInstance Entity

@Entity
public class SagaInstance {
  UUID id;
  SagaType sagaType;         // POLICY_ISSUANCE, CLAIM_PROCESSING
  SagaStatus sagaStatus;     // STARTED, IN_PROGRESS, COMPLETED,
                             // COMPENSATING, COMPENSATED
  String correlationId;      // policyId or claimId
  String currentStep;
  String failureReason;
  String payload;

  @OneToMany(cascade = ALL, orphanRemoval = true)
  List<SagaStep> steps;

  // UNIQUE(correlation_id, saga_type) → idempotent start
}

Compensation Logic

private void failSaga(SagaInstance saga) {
  saga.setSagaStatus(COMPENSATING);

  // Emit compensating commands in REVERSE order
  saga.getSteps().stream()
    .filter(s -> s.getStatus() == COMPLETED)
    .sorted(Comparator.comparingInt(
        SagaStep::getStepOrder).reversed())
    .forEach(step -> {
      String topic = COMPENSATION_TOPICS
        .get(step.getStepName());
      if (topic != null) {
        kafkaTemplate.send(topic,
          saga.getCorrelationId(),
          saga.getPayload());
      }
    });
}
Idempotency: A UNIQUE constraint on (correlation_id, saga_type) prevents duplicate saga starts — safe to retry startPolicyIssuanceSaga() multiple times.

Optimistic Locking

🔒
Applied to: All mutable JPA entities in all services
Prevents lost updates in concurrent modification scenarios 
// Every mutable entity has this field
@Entity
public class Customer {
  // ... other fields ...

  @Version
  private Long version;  // JPA increments this on every UPDATE
                          // If version doesn't match, throws OptimisticLockException
}

// Generated SQL: UPDATE customers SET ... WHERE id=? AND version=?
// If 0 rows updated → concurrent modification detected → rollback
Interview Insight: Optimistic locking is better than pessimistic (SELECT FOR UPDATE) for read-heavy workloads. It only adds overhead on actual conflicts. For event sourcing, the UNIQUE(policy_id, event_version) constraint serves the same purpose.

Scoring Rules

// FraudScoringEngine - target < 200ms
int score = 0;

// Rule 1: High claim velocity (30d)
if (customerClaims30d >= 3) score += 25;

// Rule 2: Address claim concentration (90d)
if (addressClaims90d >= 2) score += 20;

// Rule 3: Broker claim velocity (7d)
if (brokerClaims7d >= 10) score += 15;

// Rule 4: Weekend / holiday loss date
if (isWeekendOrHoliday(lossDate)) score += 5;

// Rule 5: Multiple claims in 7d
if (customerClaims7d >= 2) score += 20;

// Cap at 100. CRITICAL (>= 80) → auto-open fraud case
score = Math.min(score, 100);

👁️

// ClaimsService.approveClaim()
public void approveClaim(String claimId, BigDecimal approvedAmount, String approvedBy) {
  Claim claim = claimRepo.findByClaimId(claimId).orElseThrow();

  if (approvedAmount.compareTo(new BigDecimal("10000")) > 0) {
    // Log that supervisor approval is required
    // RBAC enforced at API layer via @PreAuthorize("hasRole('SUPERVISOR')")
    log.info("High-value claim {} requires supervisor approval for amount {}",
             claimId, approvedAmount);
  }

  claim.setStatus(APPROVED);
  claim.setPaidAmount(approvedAmount);
  claimRepo.save(claim);
  publishApprovedEvent(claim);
}

Interview Prep — Java 21

Q: Where are Java records used in this project?

All DTOs are Java 21 records. Records provide immutable data carriers with auto-generated constructor, getters, equals, hashCode, and toString — perfect for request/response objects that should not be mutated.

// Example DTO as a record
public record CustomerRegistrationRequest(
    String firstName,
    String lastName,
    @Email String email,
    String ssn,
    LocalDate dateOfBirth
) {}

Records reduce boilerplate (no need for Lombok @Data), are thread-safe by design, and work naturally with Jackson for JSON serialization.

Q: What Java 21 virtual threads (Project Loom) opportunities exist here?

While not explicitly in the code, Spring Boot 3.2 supports virtual threads via spring.threads.virtual.enabled=true. This would benefit the OutboxPublisher's blocking I/O (DB polling + Kafka publish), Kafka listeners, and external API calls (Stripe, SendGrid, Twilio) — all are I/O-bound and would scale better with virtual threads than platform threads.

Q: What sealed classes / pattern matching opportunities exist?

The SagaStatus enum (STARTED, IN_PROGRESS, COMPLETED, COMPENSATING, COMPENSATED) and PolicyStatus enum could be modeled as sealed interfaces with pattern matching in switch expressions:

// Modern Java 21 pattern matching in switch
String message = switch (claim.getStatus()) {
  case APPROVED -> "Claim approved for $" + claim.getPaidAmount();
  case DENIED   -> "Coverage verification failed";
  case CLOSED   -> "Claim closed";
  default       -> "Processing: " + claim.getStatus();
};
Q: How is @Transactional(propagation = MANDATORY) used in policy-service?

PolicyEventStore.appendEvent() is annotated with @Transactional(propagation = MANDATORY). This means the method requires an existing transaction — it will throw IllegalTransactionStateException if called outside one. This is a safety guarantee: the event store write must always be part of a larger business transaction so that the business entity write and the event store write are atomic. The caller (e.g., PolicyService.issuePolicy()) provides the outer @Transactional boundary.

Q: How does Spring's OAuth2 resource server validate Keycloak JWTs?

Each service is configured as an OAuth2 Resource Server. On startup, Spring fetches Keycloak's JWKS (public keys) from KEYCLOAK_JWK_URI. For each incoming request, Spring Security validates: (1) JWT signature using the RS256 public key, (2) expiration time, (3) issuer claim. No session storage needed — fully stateless. The JWKS URI is cached and periodically refreshed.

# application.yml

Q: How does Flyway integrate with Spring Boot and why ddl-auto: validate?

Spring Boot auto-configures Flyway when it's on the classpath. On startup, Flyway runs pending migrations in version order. ddl-auto: validate tells Hibernate to verify that the JPA entity mappings match the existing schema — but never change it. This separates concerns: Flyway owns schema evolution, JPA only talks to the DB. This prevents accidental schema drops/modifications in production.

Q: How does @Scheduled work for OutboxPublisher and payment retry?

@Scheduled(fixedDelay = 100) on OutboxPublisher means: wait 100ms after the previous execution completes before starting the next one (not a fixed rate). This prevents overlapping executions if publishing is slow. Payment retry uses fixedDelay = 300_000 (5 minutes). Both require @EnableScheduling on a configuration class. In Kubernetes with multiple pods, both will run — OutboxPublisher uses DB row locking to prevent duplicate publishes.

Q: How is Caffeine cache configured in premium-calc and reference-data?

premium-calc-service uses a 1-hour TTL cache for computed rates. reference-data-service uses a 4-hour TTL for rate tables and state rules. These are read-heavy, rarely-changing datasets — perfect for local in-process caching. Caffeine is the default cache provider in Spring Boot and supports size-based eviction and TTL.

Interview Prep — Microservices

Q: Why does each service have its own database?

Database per service is a core microservices pattern. It provides: (1) loose coupling — services can evolve their schema independently, (2) technology freedom — each service could use a different DB type, (3) failure isolation — one DB going down doesn't affect others, (4) independent scaling — the payment DB can be scaled separately. The tradeoff is no cross-service joins — you use eventual consistency via Kafka events instead.

Q: How does the platform avoid distributed transactions?

Instead of 2-phase commit (2PC), the platform uses two patterns: (1) Transactional Outbox — ensures at-least-once delivery of events without distributed transactions, (2) Saga Pattern — replaces ACID transactions with a sequence of local transactions + compensating transactions on failure. This trades strong consistency for eventual consistency with explicit rollback semantics.

Q: How is the audit trail implemented? Why is it separate from domain services?

audit-service (cg-audit consumer group) subscribes to all domain Kafka topics. Every domain event automatically flows into the audit log without any domain service needing to know about audit. This is the Event-Carried State Transfer pattern. The audit table is append-only — no updates or deletes. This separation means adding audit requirements never touches business logic.

Q: What's the difference between the outbox pattern and a choreography saga?

Outbox pattern solves the dual-write problem for a single service publishing events reliably. Choreography saga has each service react to events and emit the next event — no central coordinator. Orchestration saga (what this platform uses) has a central workflow-orchestrator that commands each service explicitly. Orchestration is easier to debug (saga state in DB) but adds a dependency; choreography is more decoupled but harder to track flows.

Q: How does the platform handle service discovery in Kubernetes?

In Kubernetes, services communicate via Kubernetes DNS (e.g., customer-service.ns-tier0.svc.cluster.local). This replaces Eureka/Consul for service discovery. Kafka brokers are referenced by their internal DNS names (kafka-1:9092). The Keycloak JWKS endpoint is resolved via the keycloak service DNS name.

Interview Prep — Kafka

Q: Why acks=all + enable.idempotence=true + retries=3?

acks=all (equivalent to acks=-1): the producer waits for acknowledgment from all in-sync replicas (ISRs), not just the leader. Combined with min.insync.replicas=2, this means at least 2 of 3 replicas must have the message before the producer considers it sent. enable.idempotence=true assigns a producer ID + sequence number to each message, allowing the broker to deduplicate retried messages (exactly-once producer semantics). retries=3 handles transient leader elections. Together: no data loss, no duplicates from the producer side.

Q: Why disable auto-commit and use AckMode.RECORD?

With enable-auto-commit=false + AckMode.RECORD, the offset is committed only after the message processing code returns successfully. If the service crashes mid-processing, the offset is not committed, so the message will be redelivered to another instance. This gives at-least-once delivery semantics. With auto-commit, you could commit the offset before processing completes, losing the message on crash (at-most-once). The tradeoff is that consumers must be idempotent — the same message might be processed twice.

Q: Why do high-volume topics have 24 partitions?

Partitions are the unit of parallelism in Kafka. With 24 partitions and concurrency=3 per consumer pod, you need 8 pods to fully consume a topic in parallel (24/3=8). High-volume topics (policy.policies.issued, payment.payments.*, claims.claims.*) are expected to have high throughput, so more partitions allow horizontal scaling. Low-volume topics (audit, reference data) use 3-6 partitions since throughput requirements are lower.

Q: Why use KRaft mode instead of ZooKeeper?

KRaft (Kafka Raft) is the new consensus mechanism that eliminates ZooKeeper as a dependency. Benefits: (1) simpler deployment — no separate ZooKeeper ensemble, (2) faster controller failover, (3) supports more partitions per cluster, (4) ZooKeeper is deprecated as of Kafka 3.5. This platform uses KRaft with 3 nodes where each node acts as both broker and controller.

Q: How does the audit-service consume ALL topics without missing events?

audit-service uses its own consumer group cg-audit which maintains its own offset per partition. Each consumer group gets a full, independent copy of every message. The auto-offset-reset: earliest setting ensures that if the audit service restarts with a new group ID, it reads from the beginning. The append-only audit table means duplicate processing is safe (idempotent inserts).

Interview Prep — System Design

Q: How would you scale the payment-service to handle 10x traffic?

Horizontal scaling: HPA already configured (max 10 pods). Kafka consumer concurrency=3 per pod × 10 pods = 30 parallel consumers for 24-partition topic. DB bottleneck: HikariCP max-pool=20 per pod × 10 pods = 200 connections to one PostgreSQL — this becomes the bottleneck. Solutions: (1) PgBouncer connection pooler between service and DB, (2) read replicas for status queries, (3) partition the payments table by payment_date or customerId for true horizontal DB scaling.

Q: What happens if the workflow-orchestrator crashes mid-saga?

The saga state is persisted in the saga_instances and saga_steps tables. On restart, the orchestrator can query for IN_PROGRESS sagas and resume them. The current step's command topic can be re-published — services must be idempotent (payment uses idempotency key, policy issue is guarded by unique policyId). The unique constraint on (correlation_id, saga_type) prevents starting duplicate sagas.

Q: How does the system prevent the "thundering herd" problem on startup?

The readinessProbe at /actuator/health/readiness with initialDelaySeconds: 30 keeps pods out of the load balancer until they're ready. For OutboxPublisher, the 100ms poll interval naturally spreads DB load. Multiple pods polling the outbox simultaneously could cause duplicate publishes — this is handled by the status check (only PENDING events are processed) and ideally a SELECT FOR UPDATE SKIP LOCKED pattern to claim events exclusively.

Q: How would you design the fraud detection for real-time performance?

The current design achieves <200ms by: (1) Redis INCR — O(1) atomic counter operations, (2) rule-based scoring — no ML model inference, (3) pre-computed counters — counters are maintained incrementally on each claim filed, not computed on-demand. To scale further: Redis Cluster for counter sharding, Kafka Streams for windowed aggregations, feature store for ML features. For ML, the scoring engine is designed as a pluggable component.

Q: How is backward compatibility maintained for Kafka message schemas?

Avro schemas with a Schema Registry enforce schema evolution rules. The platform uses BACKWARD compatibility by default — new consumers can read messages written by old producers. Rules: (1) new fields must have defaults (e.g., "default": null for optional fields), (2) never remove fields without deprecation, (3) never change field types. In the Avro schemas shown, all optional fields use union type ["null", "string"] with "default": null.

Q: How would you implement GDPR right-to-erasure in this microservices architecture?

The customer-service implements soft delete: requestGdprErasure() anonymizes PII fields (name → "REDACTED", email → anonymized hash, ssnHash cleared), sets deleted_at timestamp, and emits customer.customers.gdpr-erased event. Downstream services (notification, analytics) consume this event and purge their own copies of customer data. This is the event-driven data erasure pattern — each service is responsible for its own PII. Kafka logs are a challenge: compaction with null-value tombstone records is used to erase from the log.

Interview Prep — API Design

Q: How does the platform follow REST best practices?

Versioning: All endpoints use /api/v1/ prefix. Resource naming: plural nouns (/customers, /policies, /claims). HTTP verbs: POST for create, GET for read, PUT for full update, PATCH for partial, DELETE for erasure. Status codes: 201 Created for POST, 404 with ProblemDetail for not-found, 409 Conflict for duplicates, 400 for validation failures. Error format: RFC 7807 ProblemDetail with machine-readable type URIs.

Q: How is API security designed for multi-role access?

Keycloak manages roles: CUSTOMER, ADMIN, ADJUSTER, SUPERVISOR, BROKER, COMPLIANCE, SYSTEM. Each endpoint declares its role requirements via @PreAuthorize. The SYSTEM role is for service-to-service calls (e.g., KYC callback from external provider). Role hierarchy means ADMIN can do anything CUSTOMER can do. Claims requiring supervisor approval for >$10K use the SUPERVISOR role — an ADJUSTER JWT cannot call approve on high-value claims.

Q: How would you add API rate limiting?

Resilience4j already has a RateLimiter module. At the API Gateway layer (not yet implemented), rate limiting by customer ID or IP would be more effective. For the fraud scoring endpoint (<200ms SLA), rate limiting protects against abuse. A Redis-based token bucket (similar to the velocity counters pattern already in use) would be idiomatic here, with keys like ratelimit:{customerId}:{endpoint}:{minute} and INCR + EXPIRE.

Q: How does the payment idempotency key work end-to-end?

The caller (workflow-orchestrator) generates a UUID as idempotency key and includes it in the payment command. The payment-service first checks if a payment with that key exists (findByIdempotencyKey()). If found, it returns the existing result without calling Stripe again. If the key is new, it creates the payment, calls Stripe, and stores the result. The UNIQUE constraint at DB level ensures no race condition — two concurrent requests with the same key will result in one succeeding and one getting a constraint violation, which is caught and returns the existing record.
